Describes measures needed to roll out the pool interface in the field, such as provisioning and over-the-air updates.

Rolling out a device to multiple sites introduces challenges. For instance, the WiFi credentials can no longer be part of the source code; applying firmware updates or accessing crash information over USB becomes cumbersome; and even simply finding the IP address of the device on the LAN may create a challenge.

To address such challenges we use a two tiered approach, in which the Bootloader decides what application to boot.

  1. the Factory application takes care of provisioning WiFi credentials, downloading the Interface application and switching to the Interface app
  2. the Interface application provides the core of the functionality, including updating itself.

The following sections describes the process in further detail. A special page is dedicated to the interface application, which provides the core functionality of the pool interface.

Bootloader

The partition table is shown in the table below. The bootloader does some minimum configuration of internal modules, and select the application partition to boot, based on the partition table and ota_data (if any).

The bootloader in a nutshell:

  • If app data is erased, it boots the factory application (assuming it is present).
  • After the first OTA update, the ota_data partition is updated to specify which OTA app slot partition should be booted next. If the application works fine, it marks itself as ESP_OTA_IMG_VALID in the ota_data. Otherwise, it marks itself ESP_OTA_IMG_INVALID.
  • It support application rollback to keep the device working after the update. It automatically rolls back to the previous version, if the application doesn't pass its self test.

Factory application

The Factory application, handles the provisioning and triggers an over-the-air (OTA) download of the interface app. It then switches to the new app.

The Factory application takes are of:

  • Provisioning WiFi credentials.
  • Connecting to the WiFi network.
  • Download the Interface application (and pass control to it).

Each of these are described in detail in the following section

Provisioning WiFi credentials

For testing purposes, WIFI_CONNECT_SSID and WIFI_CONNECT_PASSWORD can also be provisioned using `Kconfig`. If empty, the device will use credentials from flash memory.

A phone is used to connect to the factory app using Bluetooth Low Energy (BLE). The Factory application will advertise itself to the phone app. Using this phone the user specifies the WiFi credentials. During the process the phone remains in contact with the ESP while the WiFi is set up and connects to an access point. The WiFi SSID and password are stored in the nvs partition of flash memory.

You can find the source code of this app in the android directory. If you have an iOS phone, or you have problems compiling the Android app, you can use the Espressif BLE Provisioning app that is available from the Android and iOS app stores. However .. you need to change the _ble_device_name_prefix to PROV_ in factory/main/main.c, and change the config.service_uuid in factory/components/factory_ble_prov/src/factory_ble_prov.c.

Connecting to the WiFi network

To establish the WiFi connection, the code base uses the git submodule ESP32_wifi-connect. This component makes this API available as a reusable component. It connects to a WiFi Access Point, and automatically reconnects when the connection drops.

More information on this component can be found at Github.

Download the Interface application

The ESP-IDF API provides a update mechanism that allows a device to update itself based on data received while the normal firmware is running. Our code base uses the git submodule ESP32_ota-update-task that makes the API available as a component.

The location of the Factory application on the internet is specified by OTA_UPDATE_FIRMWARE_URL in Kconfig. Note that for HTTPS connections, you will need to add the server's public certificate to components/ota_update_task/CMakelists.txt, and uncomment some lines in ota_update_task.c with server_cert_pem.

It checks for an OTA update on a network server specified using Kconfig. If has a different binary it applies the update. Upon completion, the device resets to activate the downloaded code. Note that we use the term "update" loosely, because it can also be used to downgrade the firmware.

To determine if the currently running code is different as the code on the server, it compares the project name, version, date and time. Note that these are not always updated by the SDK. The best way to make sure they are updated is by committing your code to Git and building the project from scratch.
